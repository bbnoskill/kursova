РОЗДІЛ 1. АНАЛІЗ ПРЕДМЕТНОЇ ОБЛАСТІ ТА ПОСТАНОВКА ЗАВДАННЯ
1.1. Опис предметної області
Актуальність теми
У сучасних умовах стрімкого розвитку цифрового споживання музичного та аудіоконтенту потреба в автоматизованих системах управління медіатеками стає критично важливою. За відсутності спеціалізованої платформи адміністрування каталогів виконавців, альбомів та аудіозаписів здійснюється вручну, що призводить до систематичних помилок обліку, дублювання контенту та неефективного розмежування прав доступу між різними категоріями користувачів. Розробка системи AudioLib забезпечує централізоване управління медіафондом та точний облік взаємодії користувачів із контентом у реальному часі.
Визначення важливих понять
Для розуміння роботи системи варто виділити кілька ключових понять.
Аудіотека — це програмна система для структурованого зберігання, каталогізації та надання доступу до аудіоконтенту різних типів.
Аудіозапис (record) — базова одиниця медіафонду системи; може належати до трьох типів: пісня (song), подкаст (podcast) або аудіокнига (audiobook).
Підписка — тарифний план користувача, що визначає рівень доступу до контенту. Система підтримує два плани: безкоштовний (free) та преміум (premium).
Прослуховування (listening) — подія взаємодії користувача з аудіозаписом; є основною транзакційною одиницею для накопичення статистики.
Плейлист — персоналізована добірка аудіозаписів, створена конкретним користувачем.
API (Application Programming Interface) — програмний інтерфейс взаємодії між клієнтською та серверною частинами системи.
MusicBrainz — відкрита база даних музичної метаінформації, яка використовується як первинне джерело наповнення каталогу.
Сфера застосування
Система призначена для музичних стримінгових платформ та цифрових аудіотек, що обслуговують різні категорії слухачів: від безкоштовних користувачів з обмеженим доступом до преміум-підписників із повним доступом до каталогу аудіокниг.
1.2. Вимоги до обробки даних
Характеристика бізнес-процесів
Система підтримує три ключових бізнес-процеси:
Каталогізація та управління медіафондом: централізоване внесення даних про нових виконавців, завантаження аудіозаписів із прив`язкою до конкретних альбомів та жанрів, структурування бази для забезпечення багатофакторного пошуку.
Адміністрування клієнтської бази та управління підписками: реєстрація облікових записів користувачів, оформлення тарифних планів (безкоштовний/преміум), моніторинг термінів дії підписок та автоматична зміна статусу доступу після їх закінчення.
Облік взаємодії з контентом та тарифікація доступу: фіксація кожного факту прослуховування аудіозапису для накопичення статистичних даних, а також перевірка прав доступу в реальному часі.
Способи накопичення та аналізу даних
Дані накопичуються у реляційній базі даних PostgreSQL. Кожна подія прослуховування фіксується в транзакційній таблиці listenings із відміткою часу та тривалістю сесії. Для аналітики використовуються збережені процедури, що обчислюють агреговану статистику за довільний часовий діапазон, та представлення, що надають актуальні зрізи даних про популярність контенту.
Форми для роботи з даними
Введення даних здійснюється через спеціалізовані форми у модальних вікнах графічного інтерфейсу. Виведення інформації представлено у вигляді таблиць із пагінацією, фільтрацією та інтерактивними картками контенту. Аналітика відображається у вигляді числових показників та діаграм розподілу.
1.3. Постановка завдання
Призначення інформаційної системи
Головним призначенням системи є комплексна автоматизація управління цифровою аудіотекою: від каталогізації медіаконтенту та контролю доступу на основі типу підписки до накопичення поведінкової аналітики та ведення журналу аудиту дій.
Ролі потенційних користувачів
Для забезпечення безпеки даних та чіткого розподілу обов’язків у системі передбачено три базові ролі:
Адміністратор — має повний доступ до всіх розділів системи, керує обліковими записами користувачів, налаштовує структуру каталогу та переглядає журнал аудиту. Це дозволяє контролювати цілісність даних та відстежувати будь-які зміни в системі.
Контент-менеджер — відповідає за ведення каталогу аудіозаписів, виконавців, альбомів та жанрів. Він додає нові записи, редагує метадані та видаляє застарілий контент.
Слухач — переглядає каталог відповідно до свого типу підписки, відтворює аудіозаписи, управляє власними плейлистами та переглядає особисту статистику прослуховувань.
Перелік функцій
Функціональність побудована за модульним принципом і охоплює такі напрямки:
1) Управління каталогом:
Ведення бази виконавців із атрибутами країни та музичного стилю.
Управління альбомами та їхнім зв’язком з виконавцями.
CRUD аудіозаписів із типізацією (song/podcast/audiobook), прив`язкою до жанрів та альбомів.
Підтримка довідника жанрів із зв’язком M:N до записів та альбомів.
2) Управління користувачами та підписками:
Реєстрація та рольова авторизація користувачів.
Керування тарифними планами (free/premium) із контролем терміну дії.
Автоматична деактивація прострочених підписок.
3) Взаємодія з контентом:
Фіксація подій прослуховування з перевіркою прав доступу в реальному часі.
Управління персональними плейлистами слухача.
4) Аналітика та аудит:
Персональна статистика прослуховувань за довільний часовий діапазон.
Платформена аналітика (топ-треки, розподіл за жанрами, статистика підписок).
Ведення журналу аудиту всіх змін даних із фіксацією автора та часу операції.
РОЗДІЛ 2. РОЗРОБКА МОДЕЛЕЙ ДАНИХ
2.1. Концептуальне проектування
В межах концептуального проєктування інформаційної системи AudioLib спроектовано реляційну модель, що складається з 11 основних таблиць. Кількість таблиць відповідає встановленій вимозі 8–12 і забезпечує повну функціональність системи. Усі сутності нормалізовані до третьої нормальної форми.
Характеристика сутностей
Сутності розподілено за функціональними групами:
Управління доступом:
user_roles — класифікатор ролей із трьома значеннями (admin, content_manager, listener).
users — облікові записи користувачів із посиланнями на підписку та роль.
Каталог контенту:
artists — виконавці та гурти з атрибутами країни та музичного стилю.
albums — альбоми із посиланням на виконавця та роком випуску.
records — аудіозаписи з атрибутами назви, тривалості (у секундах), типу (song, podcast, audiobook) та року випуску; пов’язані з виконавцем та альбомом через зовнішні ключі.
genres — довідник музичних жанрів (Rock, Hip-Hop, R&B, Pop тощо).
record_genres — асоціативна таблиця зв`язку M:N між аудіозаписами та жанрами.
album_genres — асоціативна таблиця зв`язку M:N між альбомами та жанрами.
Підписки:
subscriptions — підписки з типом (free, premium), статусом, датою завершення та вартістю.
Активність та плейлисти:
playlists — персоналізовані добірки аудіозаписів із датою створення та описом.
listenings — подія прослуховування; основна транзакційна таблиця системи, що фіксує ідентифікатор користувача, ідентифікатор запису, дату та час події (listened_at) і тривалість сесії у секундах (duration_sec).
Взаємозв`язки між поняттями
Усі зв`язки між сутностями системи спроєктовано за принципом мінімізації надлишковості та суворого забезпечення посилальної цілісності на рівні СУБД.
Класифікація зв`язків за кількісним співвідношенням:
Один до багатьох (1:N) — основний тип зв`язку в системі. Використовується для побудови ієрархій (Виконавець — Альбоми, Виконавець — Аудіозаписи, Альбом — Аудіозаписи, Підписка — Користувачі, Користувач — Плейлисти) та реєстрації подій (Користувач — Прослуховування, Аудіозапис — Прослуховування).
Багато до багатьох (M:N) — реалізовано між сутностями «Аудіозапис» та «Жанр» (через таблицю record_genres), а також між «Альбом» та «Жанр» (через album_genres). Це дозволяє одному запису або альбому одночасно належати до кількох жанрів.
Стратегії підтримки цілісності:
ON DELETE CASCADE — застосовано для залежних сутностей: при видаленні виконавця каскадно видаляються його альбоми та аудіозаписи; при видаленні аудіозапису або жанру — рядки асоціативних таблиць.
ON DELETE SET NULL — застосовано для поля album_id у таблиці records, оскільки аудіозапис може існувати поза альбомом як сингл.
ON DELETE RESTRICT — застосовано неявно для довідкових таблиць genres та user_roles.
Концептуальна ER-діаграма наведена у графічному матеріалі, що додається до звіту.

2.2. Вимоги до системи накопичення даних
Система проєктується для роботи в умовах безперервного накопичення подій прослуховувань. Загальний приріст обсягу даних визначається насамперед інтенсивністю використання транзакційної таблиці listenings.
Критичні запити
Найбільш ресурсомісткими та частотними вибірками є:
Агрегація статистики прослуховувань: потребує об’єднання таблиць listenings, records та artists за часовим діапазоном.
Формування топ-рейтингів: агрегація за кількістю прослуховувань із сортуванням по всьому каталогу.
Аналітика платформи: групування прослуховувань за жанрами, розподіл підписок, щоденна активність.
Розмежування прав доступу
Доступ до даних регулюється на рівні бізнес-логіки прикладного сервера:
Адміністратор має повний доступ до всіх розділів, включаючи журнал аудиту та управління користувачами.
Контент-менеджер має доступ до CRUD каталогу (виконавці, альбоми, записи), але не до журналу аудиту та управління підписками.
Слухач має доступ лише до власних плейлистів, особистої статистики та перегляду каталогу відповідно до типу підписки.
2.3. Логічне проектування бази даних
На етапі логічного проєктування визначено остаточний склад таблиць, їхні атрибути, типи даних та обмеження цілісності. Як СУБД обрано PostgreSQL.
Для зберігання перелічуваних станів використано типи ENUM:
record_type — типи аудіозаписів (song, podcast, audiobook).
subscription_type — типи підписок (free, premium).
subscription_status — статуси підписки (active, inactive).
role_name — назви ролей (admin, content_manager, listener).
Опис таблиць з атрибутами та обмеженнями
Таблиця user_roles — класифікатор ролей.
id (SERIAL) — PRIMARY KEY.
name (ENUM role_name) — UNIQUE, NOT NULL.
description (TEXT) — —.

Таблиця subscriptions — тарифні плани.
id (SERIAL) — PRIMARY KEY.
subscription_type (ENUM subscription_type) — NOT NULL, DEFAULT free.
status (ENUM subscription_status) — NOT NULL, DEFAULT active.
valid_until (DATE) — —.
price (NUMERIC(10,2)) — NOT NULL, DEFAULT 0.
created_at (TIMESTAMP) — NOT NULL, DEFAULT NOW().

Таблиця users — облікові записи.
id (SERIAL) — PRIMARY KEY.
first_name (VARCHAR(100)) — NOT NULL.
last_name (VARCHAR(100)) — NOT NULL.
email (VARCHAR(255)) — UNIQUE, NOT NULL.
created_at (TIMESTAMP) — NOT NULL, DEFAULT NOW().
subscription_id (INTEGER) — FK → subscriptions.id, ON DELETE SET NULL.
role_id (INTEGER) — FK → user_roles.id, NOT NULL, ON DELETE RESTRICT.
is_deleted (BOOLEAN) — NOT NULL, DEFAULT FALSE.

Таблиця artists — виконавці.
id (SERIAL) — PRIMARY KEY.
name (VARCHAR(255)) — NOT NULL.
country (VARCHAR(100)) — —.
style (VARCHAR(100)) — —.

Таблиця albums — альбоми.
id (SERIAL) — PRIMARY KEY.
title (VARCHAR(255)) — NOT NULL.
release_year (INTEGER) — —.
artist_id (INTEGER) — FK → artists.id, NOT NULL, ON DELETE CASCADE.

Таблиця records — аудіозаписи.
id (SERIAL) — PRIMARY KEY.
title (VARCHAR(255)) — NOT NULL.
length (INTEGER) — NOT NULL, DEFAULT 0 (секунди).
type (ENUM record_type) — NOT NULL, DEFAULT song.
release_year (INTEGER) — —.
artist_id (INTEGER) — FK → artists.id, NOT NULL, ON DELETE CASCADE.
album_id (INTEGER) — FK → albums.id, ON DELETE SET NULL.

Таблиця genres — довідник жанрів.
id (SERIAL) — PRIMARY KEY.
name (VARCHAR(100)) — UNIQUE, NOT NULL.

Таблиця record_genres — зв`язок M:N записи ↔ жанри.
record_id (INTEGER) — PK, FK → records.id, ON DELETE CASCADE.
genre_id (INTEGER) — PK, FK → genres.id, ON DELETE CASCADE.

Таблиця album_genres — зв`язок M:N альбоми ↔ жанри.
album_id (INTEGER) — PK, FK → albums.id, ON DELETE CASCADE.
genre_id (INTEGER) — PK, FK → genres.id, ON DELETE CASCADE.

Таблиця playlists — персоналізовані добірки.
id (SERIAL) — PRIMARY KEY.
title (VARCHAR(255)) — NOT NULL.
created_at (TIMESTAMP) — NOT NULL, DEFAULT NOW().
description (TEXT) — —.
user_id (INTEGER) — FK → users.id, NOT NULL, ON DELETE CASCADE.

Таблиця playlist_records — зв`язок M:N плейлисти ↔ записи.
playlist_id (INTEGER) — PK, FK → playlists.id, ON DELETE CASCADE.
record_id (INTEGER) — PK, FK → records.id, ON DELETE CASCADE.

2.4. Реалізація процедур бізнес-логіки
Основна бізнес-логіка системи реалізована на рівні СУБД через тригери, збережені процедури та представлення. Це гарантує цілісність даних незалежно від клієнтського застосунку та забезпечує атомарність складних операцій.
Тригери
trg_check_subscription_access — тригер BEFORE INSERT на таблиці listenings. При спробі зафіксувати прослуховування перевіряє наявність активної підписки у користувача та відповідність її типу до запитуваного контенту. Аудіокниги доступні лише для преміум-підписки; у разі порушення генерує виняток із повідомленням.
trg_deactivate_expired_subscriptions — тригер BEFORE INSERT OR UPDATE на таблиці subscriptions. Автоматично встановлює статус підписки як inactive, якщо значення поля valid_until менше за поточну дату. Запобігає ситуаціям, коли прострочена підписка залишається активною через відсутність явного оновлення.
trg_audit_log — тригер AFTER INSERT OR UPDATE OR DELETE на основних таблицях каталогу та користувачів (users, records, artists, albums, subscriptions). Автоматично записує кожну зміну даних у таблицю audit_log із фіксацією автора операції, типу дії, назви таблиці та ідентифікатора запису.
Збережені процедури
sp_get_user_stats(p_user_id INT, p_days INT) — процедура, що повертає агреговану статистику прослуховувань конкретного користувача за вказану кількість останніх днів: загальну кількість прослуховувань, загальний час у секундах, топ-5 виконавців та топ-5 аудіозаписів за частотою відтворення.
sp_get_top_records(p_days INT, p_limit INT) — процедура для формування аналітичного звіту. Повертає найпопулярніші аудіозаписи за кількістю прослуховувань у вказаний часовий діапазон із зазначенням виконавця, жанрів та загального часу відтворення.
Представлення (Views)
v_active_users — показує лише користувачів із поточною активною підпискою. Використовується у запитах статистики та модулі управління підписками.
v_record_popularity — надає агреговану інформацію про кількість прослуховувань та загальний час відтворення кожного аудіозапису з прапорцем популярності (більше 100 прослуховувань за останні 30 днів).
2.5. Опис засобів інтеграції
Обмін даними між клієнтською та серверною частинами системи здійснюється за протоколом HTTP у форматі JSON. Це забезпечує легкість парсингу та сумісність з широким колом клієнтів.
Як первинне джерело наповнення каталогу використовується інтеграція з публічним API MusicBrainz за адресою https://musicbrainz.org/ws/2. Джерело надає структуровані JSON-відповіді з метаданими виконавців, альбомів та треклистів. Через обмеження швидкості (1 запит/сек) процес імпорту виконується асинхронно і може тривати до 30 хвилин.

Для реалізації інформаційної системи AudioLib обрано класичну трирівневу архітектуру:
Рівень даних — СУБД PostgreSQL (версія ≥ 14). Окрім зберігання, цей рівень реалізує частину бізнес-логіки через тригери та збережені процедури, що гарантує цілісність даних незалежно від зовнішніх підключень.
Рівень логіки — реалізований на платформі Node.js з використанням фреймворку Express. Цей рівень виконує роль REST API: обробляє HTTP-запити від клієнта, здійснює просту сесійну автентифікацію за email та взаємодіє з базою даних через пул з`єднань (node-postgres).
Рівень презентації — реалізований як односторінковий застосунок (SPA) на чистому HTML/CSS/JavaScript без зовнішніх фреймворків (AudioLib_Interface.html). Це забезпечує швидке завантаження та просту розгортку.

Логіка доступу до контенту: при кожному записі прослуховування (POST /api/listenings) тригер trg_check_subscription_access перевіряє відповідність типу підписки та типу запису, автоматично відхиляючи некоректні запити на рівні СУБД.
Аналітика прослуховувань: ендпоінт GET /api/stats/user/:id викликає процедуру sp_get_user_stats(), яка повертає агреговані JSON-дані про топ-виконавців та топ-записи за обраний діапазон.
Підписки: ендпоінт PUT /api/subscriptions оновлює запис підписки, після чого тригер trg_deactivate_expired_subscriptions автоматично перевіряє дату закінчення та коригує статус.
Дашборд: ендпоінт GET /api/dashboard виконує паралельні агрегаційні запити через Promise.all(), використовуючи каскадну логіку fallback (7 днів → 30 днів → за весь час) для гарантованого повернення топ-треків.

РОЗДІЛ 4. ОПРАЦЮВАННЯ ДАНИХ
4.1. Характеристика первинного джерела даних
Для наповнення музичного каталогу системи використано публічне API MusicBrainz (https://musicbrainz.org/ws/2). Це відкрита база метаінформації про музикантів, альбоми та записи, що надає дані у форматі JSON.
Структура отриманих даних включає:
Метадані виконавців: ім’я, країна, теги (жанри).
Метадані релізів-груп (альбомів): назва, рік першого випуску.
Метадані окремих релізів та треклистів: назви треків, тривалість у мілісекундах.
Загальний обсяг вибірки: не менше 200 виконавців, 300 альбомів та 1000 аудіозаписів у чотирьох жанрах (Hip-Hop, Rock, Pop, R&B). Використання даних відкритої бази забезпечує достовірність та реалістичність каталогу для тестування всіх модулів системи.
4.2. Алгоритм трансформації даних при завантаженні
Процес перенесення даних із зовнішнього API реалізовано в модулі import-musicbrainz.js за такою послідовністю:
Завантаження виконавців: для кожного цільового жанру виконуються пагіновані запити до ендпоінту /artist із фільтрацією за тегами. Дані зберігаються до таблиці artists із полями name, country та style.
Завантаження альбомів: для кожного збереженого виконавця виконується запит до /release-group?artist={mbid}&type=album. Дані зберігаються до таблиці albums із прив`язкою до artist_id та роком першого випуску.
Завантаження треклистів: для кожного альбому виконується запит до /release?release-group={id}&inc=recordings для отримання треків. Тривалість у мілісекундах конвертується у секунди. Дані записуються до таблиці records.
Нормалізація жанрів: теги виконавців з API порівнюються зі списком стандартних жанрів у таблиці genres. При збігу створюється зв`язок у record_genres та album_genres.
Захист від дублювання: для кожного запису використовується механізм ON CONFLICT DO NOTHING, що забезпечує ідемпотентність повторних запусків імпорту.
Дотримання rate-limit API (1 запит/сек) реалізовано через функцію sleep() між запитами.
4.3. Алгоритм генерації тестових даних
Для демонстрації повноцінної роботи системи після імпорту музичного каталогу модуль генерує тестові дані для соціальних та транзакційних таблиць:
Генерація 50+ тестових користувачів різних ролей (admin, content_manager, listener) із підписками типів free та premium, дати закінчення яких розподілені в межах від 6 місяців тому до 1 року вперед.
Генерація плейлистів: для кожного слухача стохастично створюється від 1 до 3 плейлистів із випадковою добіркою записів з реального каталогу.
Генерація 5000 подій прослуховувань (listenings): для кожної події випадково обирається користувач та запис із каталогу, часова мітка listened_at рівномірно розподіляється в межах останніх 365 днів, тривалість — від 30 до повної тривалості треку.
Завдяки такому підходу таблиця listenings наповнюється реалістичними даними, що дозволяє продемонструвати роботу всіх аналітичних модулів та представлень v_record_popularity.
4.4. Опис додаткових методів обробки даних
Додатково в системі реалізовано такі методи динамічної обробки інформації:
Контроль доступу в реальному часі: перед кожним збереженням події прослуховування тригер trg_check_subscription_access синхронно перевіряє тип підписки та тип контенту на рівні СУБД, забезпечуючи нульову можливість обходу обмежень через прямі SQL-запити.
Механізм аудиту: при будь-якій зміні даних через тригер fn_audit_log() у таблицю audit_log записується ідентифікатор поточного користувача сесії (зчитується через функцію PostgreSQL current_setting('app.current_user_id')), тип операції та ідентифікатор зміненого запису.
Агрегація статистики: оскільки дані про прослуховування накопичуються рядково, для виведення агрегованих показників використовуються збережені процедури sp_get_user_stats() та sp_get_top_records(), що обчислюють суми та рейтинги за довільний часовий діапазон за один SQL-запит.
РОЗДІЛ 5. ФУНКЦІОНАЛЬНІ МОЖЛИВОСТІ СИСТЕМИ
5.1. Інтерфейси управління даними
Інтерфейс системи побудований за модульним принципом, що забезпечує швидкий доступ до ключових функцій аудіотеки. Основна навігація здійснюється через фіксовану бічну панель, а вміст сторінок автоматично адаптується відповідно до поточної ролі користувача.
Каталог аудіозаписів є основним інструментом для перегляду та пошуку контенту. Дані подано у вигляді таблиці з підтримкою пагінації та багатокритеріальної фільтрації за виконавцем, жанром, роком випуску та типом запису. При виборі конкретної позиції відкривається детальна картка з повною інформацією, пов’язаними жанрами, виконавцем та альбомом. Контент-менеджер та адміністратор бачать додаткові кнопки редагування та видалення. Інтерфейс каталогу наведено на рис. Д.1, Д.2.
Модуль виконавців та альбомів забезпечує перегляд та управління каталогом. На загальній сторінці виконавці представлені у вигляді карток із назвою, країною та музичним стилем. Сторінка конкретного виконавця відображає всі його альбоми та записи. Інтерфейс виконавців наведено на рис. Д.3, Д.4.
Модуль плейлистів доступний слухачам і дозволяє створювати власні добірки аудіозаписів. Інтерфейс відображає список плейлистів поточного користувача з датою створення та кількістю треків. При відкритті плейлиста відображається повний перелік записів із можливістю їх відтворення, зміни порядку або видалення. Інтерфейс плейлистів наведено на рис. Д.5.
5.2. Панель показників бізнес-процесу
Центральний екран системи призначений для оперативного моніторингу ключових показників платформи. У верхній частині відображаються чотири метрики: загальна кількість аудіозаписів у каталозі, кількість активних користувачів, прослуховувань за поточний місяць та кількість зареєстрованих виконавців.
Блок «Топ-5 треків тижня» показує найпопулярніші аудіозаписи за останні 7 днів. Завдяки каскадній логіці fallback, якщо даних за 7 днів недостатньо, система автоматично розширює діапазон до 30 днів або до всього часу існування бази.
Діаграма розподілу прослуховувань за жанрами надає наочне уявлення про структуру попиту. Блок статистики підписок відображає співвідношення преміум та безкоштовних користувачів. Головна панель системи наведена на рис. Д.6.
5.3. Засоби аналітичного представлення даних
Вкладка статистики надає персоналізований огляд активності слухача. Відображаються ключові показники за обраний часовий діапазон: загальна кількість прослуховувань, сумарний час відтворення, блоки топ-5 виконавців та топ-5 треків. Реалізація базується на виклику збереженої процедури sp_get_user_stats() з параметром p_days, що встановлюється через випадаючий список діапазону.
Адміністратор та контент-менеджер бачать агреговану статистику по всій платформі через ендпоінт GET /api/stats/platform, що надає розбивку за типами контенту та щоденну активність за останні 14 днів. Інтерфейс панелі статистики наведено на рис. Д.7.
5.4. Опис додаткових реалізованих функцій
Як додаткову функцію в системі реалізовано повноцінний журнал аудиту, доступний виключно адміністратору. Журнал містить хронологічний список операцій із зазначенням автора, часу, типу дії та назви таблиці. Реалізовано фільтрацію журналу за типом операції та назвою таблиці, а також функцію «Зріз за період» для завантаження записів у заданому діапазоні дат.
Адмін-панель надає інструмент управління обліковими записами користувачів (створення, редагування ролей та активності) та запуску імпорту з MusicBrainz API з відображенням прогресу у реальному часі через Server-Sent Events або регулярний polling. Інтерфейси журналу аудиту та адмін-панелі наведено на рис. Д.8, Д.9.
ВИСНОВКИ
У ході виконання курсової роботи розроблено інформаційну систему управління музичною аудіотекою AudioLib, яка автоматизує основні процеси: каталогізацію аудіоконтенту, управління підписками, облік взаємодії користувачів із контентом та накопичення аналітичних даних.
Спроєктовано нормалізовану реляційну базу даних на основі СУБД PostgreSQL, що складається з 11 основних таблиць та 2 системних (audit_log, playlist_records). Реалізовано бізнес-логіку на рівні бази даних: три тригери (контроль доступу, деактивація підписок, аудит), дві збережені процедури для аналітики та два представлення для агрегованих зрізів. Визначено п’ять індексів для оптимізації найчастотніших запитів системи.
Розроблено серверний REST API на платформі Node.js/Express із рольовим розмежуванням доступу та 18 ендпоінтами. Реалізовано клієнтський SPA без зовнішніх фреймворків з адаптацією інтерфейсу відповідно до ролі користувача. Систему наповнено реальними даними з публічного API MusicBrainz — завантажено понад 1000 аудіозаписів, 200 виконавців та 300 альбомів у чотирьох жанрах, а також згенеровано 5000 тестових прослуховувань для перевірки аналітичних модулів.
Встановлені недоліки
Відсутність повноцінної JWT-автентифікації: поточна реалізація здійснює аутентифікацію за email без перевірки пароля та без токена, що не відповідає вимогам продакшн-середовища.
Відсутність серверної пагінації для великих каталогів: при кількості записів понад 10 000 запити до таблиці records можуть суттєво уповільнюватися.
Обмежена фільтрація в журналі аудиту: реалізовано фільтрацію за типом операції та таблицею, але відсутній пошук за конкретним ідентифікатором запису.
Перспективи покращення
Впровадження JWT-автентифікації з механізмом оновлення токенів (refresh tokens) для підвищення безпеки.
Реалізація серверної пагінації з параметрами page та limit для всіх ендпоінтів каталогу.
Додавання повнотекстового пошуку по каталогу з використанням індексів PostgreSQL tsvector для пришвидшення пошукових запитів.
Розширення аудиту з використанням типу JSONB для зберігання знімків стану рядка «до» та «після» зміни.

-- AudioLib — Схема бази даних CREATE TYPE record_type AS ENUM ('song', 'podcast', 'audiobook'); CREATE TYPE subscription_type AS ENUM ('free', 'premium'); CREATE TYPE subscription_status AS ENUM ('active', 'inactive'); CREATE TYPE role_name AS ENUM ('admin', 'content_manager', 'listener');  CREATE TABLE user_roles (     id    SERIAL PRIMARY KEY,     name  role_name NOT NULL UNIQUE,     description TEXT );  CREATE TABLE subscriptions (     id                SERIAL PRIMARY KEY,     subscription_type subscription_type NOT NULL DEFAULT 'free',     status            subscription_status NOT NULL DEFAULT 'active',     valid_until       DATE,     price             NUMERIC(10, 2) NOT NULL DEFAULT 0,     created_at        TIMESTAMP NOT NULL DEFAULT NOW() );  CREATE TABLE users (     id              SERIAL PRIMARY KEY,     first_name      VARCHAR(100) NOT NULL,     last_name       VARCHAR(100) NOT NULL,     email           VARCHAR(255) NOT NULL UNIQUE,     created_at      TIMESTAMP NOT NULL DEFAULT NOW(),     subscription_id INTEGER REFERENCES subscriptions(id) ON DELETE SET NULL,     role_id         INTEGER NOT NULL REFERENCES user_roles(id) ON DELETE RESTRICT,     is_deleted      BOOLEAN NOT NULL DEFAULT FALSE );  CREATE TABLE artists (     id      SERIAL PRIMARY KEY,     name    VARCHAR(255) NOT NULL,     country VARCHAR(100),     style   VARCHAR(100) );  CREATE TABLE albums (     id           SERIAL PRIMARY KEY,     title        VARCHAR(255) NOT NULL,     release_year INTEGER,     artist_id    INTEGER NOT NULL REFERENCES artists(id) ON DELETE CASCADE );  CREATE TABLE records (     id           SERIAL PRIMARY KEY,     title        VARCHAR(255) NOT NULL,     length       INTEGER NOT NULL DEFAULT 0,     type         record_type NOT NULL DEFAULT 'song',     release_year INTEGER,     artist_id    INTEGER NOT NULL REFERENCES artists(id) ON DELETE CASCADE,     album_id     INTEGER REFERENCES albums(id) ON DELETE SET NULL );  CREATE TABLE genres (     id   SERIAL PRIMARY KEY,     name VARCHAR(100) NOT NULL UNIQUE );  CREATE TABLE record_genres (     record_id INTEGER NOT NULL REFERENCES records(id) ON DELETE CASCADE,     genre_id  INTEGER NOT NULL REFERENCES genres(id) ON DELETE CASCADE,     PRIMARY KEY (record_id, genre_id) );  CREATE TABLE album_genres (     album_id INTEGER NOT NULL REFERENCES albums(id) ON DELETE CASCADE,     genre_id INTEGER NOT NULL REFERENCES genres(id) ON DELETE CASCADE,     PRIMARY KEY (album_id, genre_id) ); -- Тригер 1: перевірка доступу до контенту CREATE OR REPLACE FUNCTION fn_check_subscription_access() RETURNS TRIGGER AS $$ DECLARE     v_sub_type   subscription_type;     v_sub_status subscription_status;     v_record_type record_type; BEGIN     SELECT s.subscription_type, s.status INTO v_sub_type, v_sub_status     FROM users u JOIN subscriptions s ON u.subscription_id = s.id     WHERE u.id = NEW.user_id AND u.is_deleted = FALSE;      IF NOT FOUND THEN         RAISE EXCEPTION 'Користувач % не має активної підписки', NEW.user_id;     END IF;     IF v_sub_status != 'active' THEN         RAISE EXCEPTION 'Підписка користувача % неактивна', NEW.user_id;     END IF;     SELECT type INTO v_record_type FROM records WHERE id = NEW.record_id;     IF v_sub_type = 'free' AND v_record_type = 'audiobook' THEN         RAISE EXCEPTION 'Аудіокниги доступні лише для Premium підписки';     END IF;     RETURN NEW; END; $$ LANGUAGE plpgsql;  CREATE TRIGGER trg_deactivate_expired_subscriptions     BEFORE INSERT OR UPDATE ON subscriptions     FOR EACH ROW EXECUTE FUNCTION fn_deactivate_expired_subscriptions();  -- Тригер 3: аудит змін даних CREATE OR REPLACE FUNCTION fn_audit_log() RETURNS TRIGGER AS $$ BEGIN     INSERT INTO audit_log (user_id, operation, table_name, record_id, details)     VALUES (         NULLIF(current_setting('app.current_user_id', true), '')::INTEGER,         TG_OP, TG_TABLE_NAME,         CASE WHEN TG_OP = 'DELETE' THEN OLD.id ELSE NEW.id END,         CASE TG_OP WHEN 'INSERT' THEN 'Створено новий запис'                    WHEN 'UPDATE' THEN 'Оновлено запис'                    WHEN 'DELETE' THEN 'Видалено запис' END     );     RETURN CASE WHEN TG_OP = 'DELETE' THEN OLD ELSE NEW END; END; $$ LANGUAGE plpgsql;  -- Процедура: статистика користувача CREATE OR REPLACE FUNCTION sp_get_user_stats(p_user_id INTEGER, p_days INTEGER DEFAULT 30) RETURNS TABLE (total_listenings BIGINT, total_seconds BIGINT,                top_artists JSON, top_records JSON) AS $$ BEGIN     RETURN QUERY SELECT         (SELECT COUNT(*) FROM listenings l          WHERE l.user_id = p_user_id            AND l.listened_at >= NOW() - (p_days || ' days')::INTERVAL),         (SELECT COALESCE(SUM(l.duration_sec), 0) FROM listenings l          WHERE l.user_id = p_user_id            AND l.listened_at >= NOW() - (p_days || ' days')::INTERVAL),         (SELECT json_agg(row_to_json(t)) FROM (              SELECT a.id, a.name, COUNT(*) AS listen_count              FROM listenings l JOIN records r ON l.record_id = r.id              JOIN artists a ON r.artist_id = a.id              WHERE l.user_id = p_user_id                AND l.listened_at >= NOW() - (p_days || ' days')::INTERVAL              GROUP BY a.id, a.name ORDER BY listen_count DESC LIMIT 5          ) t),         (SELECT json_agg(row_to_json(t)) FROM (              SELECT r.id, r.title, a.name AS artist_name, COUNT(*) AS listen_count              FROM listenings l JOIN records r ON l.record_id = r.id              JOIN artists a ON r.artist_id = a.id              WHERE l.user_id = p_user_id                AND l.listened_at >= NOW() - (p_days || ' days')::INTERVAL              GROUP BY r.id, r.title, a.name ORDER BY listen_count DESC LIMIT 5          ) t); END; $$ LANGUAGE plpgsql;
